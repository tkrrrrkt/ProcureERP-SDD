# エンティティ定義：取引先系マスタ（Party / SupplierSite / Payee / CustomerSite / ShipTo）

本ドキュメントは、取引先系マスタのエンティティ定義である。  
仕様は別紙「取引先系マスタ 仕様概要」に従う。

---

## 1. parties（取引先法人 / Party）

### 仕様の概要
- 相手方の法人そのもの（会社単位で1レコード）
- 法人番号・インボイス番号等を保持する
- SupplierSite / CustomerSite の親エンティティ
- 導線用に is_supplier / is_customer を保持する（存在判定・検索・ナビ用）
- party_code は10桁固定（英数字可）とし、手入力／自動採番／移行投入のいずれも許容するが、tenant内一意制約を必須とする
- party_code の変更は原則禁止（移行時例外のみ）とし、必要な場合は派生表示コード（supplier_code等）も同時更新する
- is_supplier / is_customer は派生フラグ（アプリケーション層で明示的に更新し、週次バッチで整合性チェック）

### エンティティ定義

### 制約条件
- UNIQUE(tenant_id, party_code)
- is_active=false は新規選択不可

### 補足事項
- Partyは「法人」の正本であり、拠点・部門・支払はSite側で表現する
- 住所・連絡先はPartyにも持つが、実務窓口はSupplierSite/CustomerSiteの値を優先する（画面仕様で明確化）
- **監査列**：created_by_login_account_id / updated_by_login_account_id は **実質必須**（社員マスタと統一）
  - DB定義：NULL許容（login_accounts未実装のため）
  - アプリケーション層：Service層で必ずuserIdを設定
  - Phase 1（MVP-1）：FK制約なし（UUID参照のみ）
  - Phase 2（将来）：login_accounts実装後にFK制約を追加
- **is_supplier / is_customer の更新ルール**：
  - SupplierSite作成時 → is_supplier = true に更新
  - SupplierSite削除時 → 有効なSupplierSiteが残っていなければ is_supplier = false に更新
  - CustomerSite作成時 → is_customer = true に更新
  - CustomerSite削除時 → 有効なCustomerSiteが残っていなければ is_customer = false に更新
  - アプリケーション層（Service層）で明示的に更新し、週次バッチで整合性チェック

## 2. supplier_sites（仕入先拠点 / SupplierSite）

### 仕様の概要
- 取引先（Party）に紐づく「購買の実務窓口」拠点
- 発注・見積・納期回答の相手（購買の主役エンティティ）
- 支払先（Payee）を参照する（未指定なら自動生成）
- 住所・連絡先は**1セットのみ**（MVP最適・推奨）

| supplier_code | varchar(50) | NO | `P000000001-0000000001` | 表示用（保持、最大21文字） |

| supplier_name_kana | varchar(200) | YES |  | 任意 |

- UNIQUE(tenant_id, party_id, supplier_sub_code)

- FK(tenant_id, party_id) -> parties(tenant_id, id)
- FK(tenant_id, payee_id) -> payees(tenant_id, id)

### 補足事項
- **Payee自動生成ロジック**（仕様MD 5章）：
  - `payee_id` 未指定時、同一 `party_id` + `supplier_sub_code` のPayeeが存在するかチェック
  - **存在する場合**：既存Payeeを紐づける（新規作成しない）
  - **存在しない場合**：新規Payee作成（`payee_sub_code` = `supplier_sub_code`）
  - 初回コピーのみ・以後独立（同期なし）（仕様MD 5.3）
  - 論理削除後の再利用：SupplierSiteを論理削除（is_active=false）後、同じsub_codeで再作成した場合、既存Payeeが再利用される
- **監査列**：created_by_login_account_id / updated_by_login_account_id は **実質必須**（社員マスタと統一、詳細はPartiesの補足事項を参照）

## 5. ship_tos（納入先 / ShipTo）

### 仕様の概要
- 得意先（CustomerSite）が指定する「実際にモノを届ける場所」
- エンドユーザー／現場／工事場所等の直送先を含む
- ShipTo は CustomerSite 配下（customer_site_id で参照）
- ただし、納入先コード（ship_to_code）は**独立した別コードとして管理**し、
  親子関係がコード上・対外上に見える形式は採用しない（10桁独立コード）
- 住所・連絡先は1セットのみ（MVP）
- **独立コード方針の理由**：
  - 直送先の柔軟な管理（CustomerSiteとの紐づけ変更に対応）
  - 納入先の機密性確保（得意先との関係を対外に見せない）
  - 将来的な購買側の直送対応（発注先から直送先への納品）

### エンティティ定義

| カラム | 型 | NULL | 例 | 補足 |

| ship_to_code | varchar(50) | NO | `0000000001` / `0A0B000001` | **10桁固定**（テナント設定）※保存時正規化／独立コード |
| ship_to_name | varchar(200) | NO | `DEF工場 第1倉庫` | 納入先名称 |
| ship_to_name_kana | varchar(200) | YES |  | 任意 |

- 自社倉庫・自社拠点は倉庫マスタで管理し、ShipToは直送先中心
- 得意先→納入先の関係は customer_site_id で担保し、外部に見せない
- **独立コード体系**：ship_to_code は CustomerSite のコードを含まない独立した10桁コード（上記「独立コード方針の理由」を参照）
- **監査列**：created_by_login_account_id / updated_by_login_account_id は **実質必須**（社員マスタと統一、詳細はPartiesの補足事項を参照）

## 6. コード正規化（共通）

保存時に以下を適用する：
- trim（前後空白除去）
- 半角化
- 英字は大文字に統一（英数字モード時）

文字種制約はテナント設定に従う：
- 数字のみ：`^[0-9]{10}$`
- 英数字：`^[0-9A-Z]{10}$`
- 論理無効化（is_active=false）後のコード再利用は禁止（復活は同一レコードを再活性化で対応）
